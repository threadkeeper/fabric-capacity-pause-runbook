Automatically pause your Microsoft Fabric capacity every evening and resume it each morning using Azure Automation. This saves costs by only running (and paying for) capacity when you need it.

> **Best practice for developer capacities:** If you are using a Fabric capacity for development or testing, scheduling it to run only during working hours (7 AM – 6 PM daily) is strongly recommended. A paused capacity incurs no compute charges, so this simple automation can cut your Fabric costs by over 50%.

## What You Will Set Up

| Component | Purpose |
|---|---|
| **Azure Automation Account** | Hosts and runs your runbooks on a schedule |
| **System-Assigned Managed Identity** | Lets the Automation Account authenticate to Azure without storing credentials |
| **Automation Variables** | Store your Subscription ID, Resource Group, and Capacity Name so they are easy to update |
| **Two Runbooks** | `Pause-FabricCapacity.ps1` (suspend) and `Resume-FabricCapacity.ps1` (resume) |
| **Two Schedules** | One to pause every day at 6 PM and one to resume every day at 7 AM |

- An Azure subscription with an existing **Microsoft Fabric capacity** already created.

- For the VS Code path: [Visual Studio Code](https://code.visualstudio.com/) and [PowerShell 7+](https://learn.microsoft.com/powershell/scripting/install/installing-powershell-on-windows) installed locally.

# Path A – Azure Portal (No Local Tools Required)

This path uses only the Azure Portal in your browser.

## Step 1 – Create an Automation Account

1. Go to the [Azure Portal](https://portal.azure.com).
2. In the top search bar, type **Automation Accounts** and select it.
3. Click **+ Create**.
4. Fill in the basics:
   - **Subscription**: Select your subscription.
   - **Resource group**: Select an existing resource group or create a new one (this is for the Automation Account itself; it does not need to be the same group as your Fabric capacity).
   - **Automation account name**: Choose a name, e.g. `aa-fabric-scheduler`.
   - **Region**: Select the same region as your Fabric capacity, or the closest available.
5. Click **Next: Advanced**.
6. Under **Identity**, ensure **System assigned** is checked (it is by default).
7. Click **Review + Create**, then **Create**.
8. Wait for the deployment to finish, then click **Go to resource**.

## Step 2 – Grant the Managed Identity Permission on the Fabric Capacity

The Automation Account's managed identity needs permission to pause and resume your Fabric capacity.

1. In the Azure Portal, navigate to your **Fabric capacity** resource:
   - Search for **Microsoft Fabric** in the top search bar → select **Fabric capacities** → click your capacity.
2. In the left menu, click **Access control (IAM)**.
3. Click **+ Add** → **Add role assignment**.
4. On the **Role** tab, select **Contributor**.
5. Click **Next**.
6. On the **Members** tab:
   - Set **Assign access to** → **Managed identity**.
   - Click **+ Select members**.
   - In the **Managed identity** dropdown, select **Automation Account**.
   - Select your Automation Account (e.g. `aa-fabric-scheduler`).
   - Click **Select**.
7. Click **Review + assign**, then **Review + assign** again.

## Step 3 – Add Automation Variables

Variables let you store your Fabric capacity details in one place. If you ever rename or move the capacity, you update the variables instead of editing every runbook.

1. In the Azure Portal, navigate to your **Automation Account**.
2. In the left menu under **Shared Resources**, click **Variables**.
3. Click **+ Add a variable** and create each of the following three variables:

| Name | Type | Value | Encrypted |
|---|---|---|---|
| `SubscriptionId` | String | Your Azure Subscription ID | No |
| `ResourceGroupName` | String | The resource group containing your Fabric capacity | No |
| `FabricCapacityName` | String | Your Fabric capacity name | No |

4. For each one:
   - Enter the **Name** exactly as shown above.
   - Set **Type** to **String**.
   - Paste the **Value**.
   - Leave **Encrypted** set to **No** (these are not secrets).
   - Click **Create**.

> **Why variables instead of hardcoded parameters?** Variables are reusable across runbooks, editable without touching code, and visible at a glance in the portal.

## Step 4 – Import Required Modules

The runbooks use the `Az.Accounts` and `Az.Fabric` PowerShell modules. You need to make sure they are available in your Automation Account.

1. In your Automation Account, click **Modules** in the left menu under **Shared Resources**.
2. Click **+ Add a module**.
3. Under **Browse from gallery**, click **Browse gallery**.
4. Search for **Az.Accounts**.
   - Select it, click **Import**.
   - Set **Runtime version** to **7.2** (recommended).
   - Click **Import**. Wait for the status to change to **Available** (this may take a few minutes).
5. Repeat the above steps for **Az.Fabric**:
   - Click **+ Add a module** → **Browse gallery** → search **Az.Fabric** → **Import** (Runtime version **7.2**).

> **Important:** `Az.Accounts` must finish importing before you import `Az.Fabric`, because `Az.Fabric` depends on it.

## Step 5 – Create the Runbooks

### 5a – Pause Runbook

1. In your Automation Account, click **Runbooks** in the left menu under **Process Automation**.
2. Click **+ Create a runbook**.
3. Fill in:
   - **Name**: `Pause-FabricCapacity`
   - **Runbook type**: **PowerShell**
   - **Runtime version**: **7.2**
   - **Description**: `Suspends the Fabric capacity to save costs.`
4. Click **Create**.
5. In the editor that opens, delete any placeholder text and paste the entire contents of [`runbooks/Pause-FabricCapacity.ps1`](runbooks/Pause-FabricCapacity.ps1) from this repository.
6. Click **Save**, then click **Publish** → **Yes**.

### 5b – Resume Runbook

1. Click **Runbooks** → **+ Create a runbook** again.
2. Fill in:
   - **Name**: `Resume-FabricCapacity`
   - **Runbook type**: **PowerShell**
   - **Runtime version**: **7.2**
   - **Description**: `Resumes the Fabric capacity for business hours.`
3. Click **Create**.
4. Paste the entire contents of [`runbooks/Resume-FabricCapacity.ps1`](runbooks/Resume-FabricCapacity.ps1).
5. Click **Save**, then click **Publish** → **Yes**.

## Step 6 – Test a Runbook (Optional but Recommended)

1. Open one of your runbooks (e.g. `Pause-FabricCapacity`).
2. Click **Test pane**.
3. Fill in the parameters:
   - **SubscriptionId**: Your subscription ID
   - **ResourceGroupName**: Your resource group name
   - **CapacityName**: Your Fabric capacity name
4. Click **Start**.
5. Watch the output. You should see messages like:
   ```
   Successfully authenticated with managed identity.
   Capacity 'fabricprod001' current state: Active
   Suspending Fabric capacity 'fabricprod001'...
   Fabric capacity 'fabricprod001' has been suspended successfully.
   ```
6. If you see errors about permissions, revisit [Step 2](#step-2--grant-the-managed-identity-permission-on-the-fabric-capacity).

## Step 7 – Create Schedules

### 7a – Pause Schedule (Every Evening)

1. In your Automation Account, click **Schedules** in the left menu under **Shared Resources**.
2. Click **+ Add a schedule**.
3. Fill in:
   - **Name**: `Pause-Daily-Evening`
   - **Description**: `Pauses Fabric capacity at 6 PM every day.`
   - **Starts**: Select today's date, set the time to **6:00 PM**.
   - **Time zone**: Select your local time zone.
   - **Recurrence**: **Recurring**
   - **Recur every**: **1 Day**
   - **Set expiration**: **No**.
4. Click **Create**.

### 7b – Resume Schedule (Every Morning)

1. Click **+ Add a schedule** again.
2. Fill in:
   - **Name**: `Resume-Daily-Morning`
   - **Description**: `Resumes Fabric capacity at 7 AM every day.`
   - **Starts**: Select tomorrow's date, set the time to **7:00 AM**.
   - **Time zone**: Select your local time zone.
   - **Recurrence**: **Recurring**
   - **Recur every**: **1 Day**
   - **Set expiration**: **No**.
3. Click **Create**.

> **Tip:** With a daily schedule, the capacity is paused every evening including weekends, ensuring you never pay for idle time. This is especially valuable for developer and test capacities.

## Step 8 – Link Schedules to Runbooks

### 8a – Link Pause Schedule

1. Go to **Runbooks** → open **Pause-FabricCapacity**.
2. In the left menu, click **Schedules** → **+ Add a schedule**.
3. Click **Link a schedule to your runbook** → select **Pause-Daily-Evening**.
4. Click **Configure parameters and run settings**. Enter:
   - **SUBSCRIPTIONID**: Your subscription ID
   - **RESOURCEGROUPNAME**: Your resource group name
   - **CAPACITYNAME**: Your Fabric capacity name
5. Click **OK**, then **OK** again.

### 8b – Link Resume Schedule

1. Go to **Runbooks** → open **Resume-FabricCapacity**.
2. Click **Schedules** → **+ Add a schedule**.
3. Select **Resume-Daily-Morning**.
4. Enter the same parameters as above.
5. Click **OK**, then **OK** again.

**You are done with Path A!** Your Fabric capacity will now automatically pause every evening at 6 PM and resume every morning at 7 AM.

# Path B – Visual Studio Code (Local Machine)

This path uses VS Code and the Azure PowerShell module on your local machine to deploy everything via scripts.

## Step 1 – Install Prerequisites

1. Install [Visual Studio Code](https://code.visualstudio.com/).
2. Install the [PowerShell extension for VS Code](https://marketplace.visualstudio.com/items?itemName=ms-vscode.PowerShell).
3. Open a PowerShell 7 terminal in VS Code (**Terminal → New Terminal**).
4. Install the Azure PowerShell modules:

```powershell
Install-Module -Name Az.Accounts -Scope CurrentUser -Force
Install-Module -Name Az.Automation -Scope CurrentUser -Force
Install-Module -Name Az.Fabric -Scope CurrentUser -Force
Install-Module -Name Az.Resources -Scope CurrentUser -Force
```

## Step 2 – Sign In to Azure

```powershell
Connect-AzAccount
```

A browser window will open. Sign in with your Azure credentials. If you have multiple subscriptions, set the correct one:

```powershell
Set-AzContext -SubscriptionId "<your-subscription-id>"
```

## Step 3 – Set Your Variables

Update these variables with your own values. These will be used in every command that follows.

```powershell
# ---- UPDATE THESE VALUES ----
$subscriptionId     = "<your-subscription-id>"
$resourceGroupName  = "<resource-group-with-fabric-capacity>"
$capacityName       = "<your-fabric-capacity-name>"

# Automation Account settings
$automationRg       = "<resource-group-for-automation-account>"  # can be the same as above
$automationName     = "aa-fabric-scheduler"
$location           = "eastus"  # choose the region closest to your Fabric capacity
```

## Step 4 – Create the Automation Account

```powershell
# Create the resource group for the Automation Account (skip if it already exists)
New-AzResourceGroup -Name $automationRg -Location $location -ErrorAction SilentlyContinue

# Create the Automation Account with a system-assigned managed identity
New-AzAutomationAccount -ResourceGroupName $automationRg `
    -Name $automationName `
    -Location $location `
    -AssignSystemIdentity
```

## Step 5 – Grant the Managed Identity Contributor Role

```powershell
# Get the managed identity's principal ID
$identity = (Get-AzAutomationAccount -ResourceGroupName $automationRg -Name $automationName).Identity.PrincipalId

# Build the Fabric capacity resource ID
$fabricResourceId = "/subscriptions/$subscriptionId/resourceGroups/$resourceGroupName/providers/Microsoft.Fabric/capacities/$capacityName"

# Assign the Contributor role on the Fabric capacity
New-AzRoleAssignment -ObjectId $identity `
    -RoleDefinitionName "Contributor" `
    -Scope $fabricResourceId
```

## Step 6 – Add Automation Variables

```powershell
# Store the Subscription ID
New-AzAutomationVariable -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Name "SubscriptionId" `
    -Value $subscriptionId `
    -Encrypted $false

# Store the Resource Group Name
New-AzAutomationVariable -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Name "ResourceGroupName" `
    -Value $resourceGroupName `
    -Encrypted $false

# Store the Fabric Capacity Name
New-AzAutomationVariable -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Name "FabricCapacityName" `
    -Value $capacityName `
    -Encrypted $false
```

> **Tip:** To update a variable later, use `Set-AzAutomationVariable` with the same parameters.

## Step 7 – Import Required Modules into the Automation Account

```powershell
# Import Az.Accounts first (dependency for Az.Fabric)
New-AzAutomationModule -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Name "Az.Accounts" `
    -ContentLinkUri "https://www.powershellgallery.com/api/v2/package/Az.Accounts" `
    -RuntimeVersion "7.2"

Write-Host "Waiting for Az.Accounts to finish importing..."
do {
    Start-Sleep -Seconds 15
    $mod = Get-AzAutomationModule -ResourceGroupName $automationRg `
        -AutomationAccountName $automationName -Name "Az.Accounts" `
        -RuntimeVersion "7.2"
} while ($mod.ProvisioningState -ne "Succeeded")
Write-Host "Az.Accounts imported."

# Import Az.Fabric
New-AzAutomationModule -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Name "Az.Fabric" `
    -ContentLinkUri "https://www.powershellgallery.com/api/v2/package/Az.Fabric" `
    -RuntimeVersion "7.2"
```

## Step 8 – Import the Runbooks

Run these commands from the root of this repository:

```powershell
# Import the Pause runbook
Import-AzAutomationRunbook -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Path ".\runbooks\Pause-FabricCapacity.ps1" `
    -Name "Pause-FabricCapacity" `
    -Type PowerShell `
    -RuntimeVersion "7.2" `
    -Published

# Import the Resume runbook
Import-AzAutomationRunbook -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Path ".\runbooks\Resume-FabricCapacity.ps1" `
    -Name "Resume-FabricCapacity" `
    -Type PowerShell `
    -RuntimeVersion "7.2" `
    -Published
```

## Step 9 – Create Schedules

```powershell
# Set your desired times (adjust as needed)
$pauseTime  = (Get-Date).Date.AddHours(18)   # 6:00 PM today
$resumeTime = (Get-Date).Date.AddDays(1).AddHours(7)  # 7:00 AM tomorrow
$timeZone   = "Eastern Standard Time"  # change to your time zone

# Create the pause schedule (every day at 6 PM)
New-AzAutomationSchedule -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Name "Pause-Daily-Evening" `
    -StartTime $pauseTime `
    -TimeZone $timeZone `
    -DayInterval 1 `
    -Description "Pauses Fabric capacity at 6 PM every day."

# Create the resume schedule (every day at 7 AM)
New-AzAutomationSchedule -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Name "Resume-Daily-Morning" `
    -StartTime $resumeTime `
    -TimeZone $timeZone `
    -DayInterval 1 `
    -Description "Resumes Fabric capacity at 7 AM every day."
```

> **Finding your time zone name:** Run `Get-TimeZone -ListAvailable | Where-Object { $_.DisplayName -like "*your city*" }` to find the correct time zone string.

## Step 10 – Link Schedules to Runbooks

```powershell
# Parameters to pass to both runbooks
$params = @{
    SubscriptionId    = $subscriptionId
    ResourceGroupName = $resourceGroupName
    CapacityName      = $capacityName
}

# Link the pause schedule
Register-AzAutomationScheduledRunbook -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -RunbookName "Pause-FabricCapacity" `
    -ScheduleName "Pause-Daily-Evening" `
    -Parameters $params

# Link the resume schedule
Register-AzAutomationScheduledRunbook -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -RunbookName "Resume-FabricCapacity" `
    -ScheduleName "Resume-Daily-Morning" `
    -Parameters $params
```

## Step 11 – Test a Runbook

```powershell
# Run the Pause runbook now to verify everything works
Start-AzAutomationRunbook -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Name "Pause-FabricCapacity" `
    -Parameters $params
```

Check the result in the Azure Portal under **Automation Account → Jobs**, or run:

```powershell
Get-AzAutomationJob -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -RunbookName "Pause-FabricCapacity" | Select-Object -First 1
```

**You are done with Path B!**

## Updating the Fabric Capacity Name Later

If you rename or switch to a different capacity, update the Automation variable:

**Portal:** Automation Account → Variables → click `FabricCapacityName` → edit the value → Save.

**PowerShell:**
```powershell
Set-AzAutomationVariable -ResourceGroupName $automationRg `
    -AutomationAccountName $automationName `
    -Name "FabricCapacityName" `
    -Value "your-new-capacity-name" `
    -Encrypted $false
```

> **Note:** If you pass the capacity name as a schedule parameter (as in the setup above), you will also need to unlink and re-link the schedule with the new parameter value. Using `Get-AutomationVariable` inside the runbook instead of parameters avoids this — see [Using Variables Inside Runbooks](#using-variables-inside-runbooks-optional) below.

## Using Variables Inside Runbooks (Optional)

Instead of passing parameters when linking schedules, you can modify the runbooks to read directly from Automation Variables. Replace the `param` block at the top of each runbook with:

```powershell
$SubscriptionId    = Get-AutomationVariable -Name "SubscriptionId"
$ResourceGroupName = Get-AutomationVariable -Name "ResourceGroupName"
$CapacityName      = Get-AutomationVariable -Name "FabricCapacityName"
```

This means you only need to update the variable in the Automation Account — no need to re-link schedules.

## Troubleshooting

| Problem | Solution |
|---|---|
| **"The client does not have authorization"** | The managed identity does not have the Contributor role on the Fabric capacity. Revisit the role assignment step. |
| **"Az.Fabric module not found"** | The module has not finished importing, or was imported under the wrong runtime version. Check Modules in the Automation Account and ensure Runtime version is **7.2**. |
| **Runbook completes but capacity is not paused** | Check the runbook output in Jobs. The capacity may already be paused, or in a transitional state. |
| **Schedule does not fire** | Verify the schedule is **Enabled**, the time zone is correct, and the start time is in the future. |
| **"Fabric capacity not found"** | Double-check the `ResourceGroupName` and `CapacityName` values. Names are case-sensitive. |
